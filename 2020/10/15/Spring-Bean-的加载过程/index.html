Spring Bean 的加载过程
发表于2020-10-15|更新于2020-10-15|JavaSpring框架
|阅读量:

Bean 的初始化概述

解析

​ 扫描 XML 或类文件，从配置或注解中获取 Bean 的定义信息，保存在BeanDefinitionMap<String , BeanDefinition> 集合中

加载

​ 通过 Bean 的定义信息来创建 Bean 的实例

Bean 的创建也会受到 作用域和依赖关系 的影响。单例作用域的 Bean 会在读取配置文件时创建 Bean 的实例。原型作用域的 Bean IOC 容器读取配置文件时不会创建 Bean 的实例，而是在调用该 Bean 时才会创建一个 Bean 的实例。

依赖关系：一个 Bean 如果有依赖，我们需要初始化依赖，然后进行关联。我们需要关注一个问题：循环依赖问题

Bean 的加载流程

一个原型作用域的 Bean 的加载会经历如下流程：

  • 获取 BeanName：对传入的 Bean 进行解析，转化为可以从 Map 中获取到的 BeanDefinition 的 BeanName。
  • 合并 Bean 的定义：对父类的定义进行合并和覆盖，如果父类还有父类，会进行递归合并，以获取完整的 Bean 定义。
  • 实例化：使用构造和工厂方法创建 Bean 的实例。
  • 属性填充：寻找并注入依赖，依赖的 Bean 还会递归调用 getBean 方法获取。
  • 初始化：调用自定义的初始化方法。
  • 获取最终的 Bean：如果是 FactoryBean 需要调用 getObject 方法，如果需要类型转换要调用 TypeConverter 进行转化。

转化 BeanName

​ BeanFactory 的 getBean 中传入的 name：

  • BeanName：可以直接获取到 BeanDefinition 中的 BeanName。通过 AbstractBeanFactory 的 doGetBean 方法中的 final String beanName = transformedBeanName(name) 进行转化。
  • Alias Name：别名。在解析阶段 alias name 和 bean name 的映射关系被注册到 SimpleAliasRegistry 中，通过该注册器获取 BeanName。
  • FactoryBean name：表示这是一个工厂 Bean，有前端修饰符 & 的，直接把前缀去掉。通过 BeanFacoryUtils 的 transformedBeanName 方法

合并 RootBeanDefinition

​ 从配置文件中读取到的 BeanDefinition 是 GenericBeanDefinition，它记录了当前 Bean 的 属性&方法的定义信息，但是对于其父类&实现接口，只用了一个 parentName&interfaceName 来记录。

  • 有继承关系：GenericBeanDefinition 的定义信息是不全的，是一个增量信息，而不是全量信息。
  • 没有继承关系：GenericBeanDefinition 的信息是完整的，可以直接转化为 RootBeanDefinition。

为了使 Bean 的定义信息完整，需要递归合并父类的定义。见 AbstractBeanFactory 的 doGetBean 方法

​ 在判断 parentName 存在的情况下，说明存在父类定义，启动合并。如果存在父类的父类，递归调用&合并。详细方法见 AbstraBeanFactory 的 getMergedBeanDefinition 方法。

​ 每次合并完父类定义进行覆盖，获取到当前类能够正确实例化的全量信息。

处理循环依赖

​ 循环依赖根据注入的时机：

  • 构造器循环依赖：依赖的对象是通过构造器传入的，发生在实例 Bean 的时候。
  • 设置循环依赖：依赖的对象是通过 setter 方式注入的，对象已经实例化，发生在属性填充和依赖注入的时候。

如果是构造器循环依赖，本质上是无法解决的。

如果是设值循环依赖，Spring 框架只支持单例下的设值循环依赖。Spring 通过对还在创建 过程中的单例，缓存并提前暴露该单例，使得其他实例可以引用该依赖。

原型模式下的循环依赖

​ Spring 不支持原型模式的任何循环依赖，检测到循环依赖会直接抛出 BeanCurrentlyInCreationException 异常。

​ Spring 使用了一个返回 ThreadLocal 线程变量的属性 protoTypesCurrentlyInCreation 。来记录当前线程正在创建的 Bean 的实例。

AbstractBeanFactory 的 protoTypesCurrentlyInCreation 属性

在 Bean 创建前记录，创建完成后删除记录 AbstractBeanFactory 的 doGetBean 方法

AbstractBeanFactory 的 beforePrototypeCreation 方法做记录操作

AbstractBeanFactory 的 afterPrototypeCreation 方法做删除操作

​ 为了节省内存空间，在单个元素时 prototypesCurrentlyInCreation 只记录 String 对象，多个依赖元素则会改用 Set 集合。

  • 构造函数依赖：比如 A 的构造函数依赖了 B。实例化 A 的阶段，匹配到要使用的构造函数，发现构造函数有参数 B，会使用 BeanDefinitionValueResolver 来检索 B 的实例。

  • 设置循环依赖：加载过程中，检索到无参构造函数，不需要检索构造参数的引用，实例化成功。到属性填充阶段 AbstractBeanFactory.populateBean，在这里进行循环依赖的注入。

    为了获得循环依赖的实例化后的引用，最终会通过检索类 dependencyDescriptor 中把以来读取出来。

无论是构造循环依赖还是设置循环依赖，在需要注入依赖的对象时，会继续去调用 BeanFactory.getBean() 去加载对象，形成一个递归。而每次调用 getBean 进行实例化前后，都使用了 prototypesCurrentlyInCreation 这个线程变量做记录，最终形成建立依赖对象的构造链。

​ 调用判定的地方在 AbstractBeanFactory.doGetBean 中，所有对象的实例化均会从这里启动。

所以在原型模式下，无论是构造循环依赖还是设置循环依赖，本质上都使用同一种方法检测，Spring 无法解决，直接抛出 BeanCurrentlyInCreation 的异常。

单例模式下的循环依赖

​ Spring 也不支持单例模式下的构造循环依赖，遇到会直接抛出 BeanCurrentlyInCreationException。和原型模式一样，单例模式也用了一个变量 defaultSingletonBeanRegistry 来记录正在创建中的 BeanName。

​ 这里会尝试往 singletonsCurrentlyInCreation 记录当前实例化的 bean。我们知道 singletonsCurrentlyInCreation 的数据结构是 Set，是不允许重复元素的，所以一旦前面 记录了，这里的 add 操作将会返回失败。

  • 单例模式的构造循环依赖

    • 加载 A。记录 singletonsCurrentlyInCreation = [a]，构造依赖 B，开始加载 B。
    • 加载 B，记录 singletonsCurrentlyInCreation = [a, b]，构造依赖 C，开始加 载 C。
    • 加载 C，记录 singletonsCurrentlyInCreation = [a, b, c]，构造依赖 A，又开 始加载 A。
    • 加载 A，执行到 DefaultSingletonBeanRegistry.beforeSingletonCreation ， singletonsCurrentlyInCreation 中 a 已经存在了，检测到构造循环依赖，直接抛出 异常结束操作。

  • 单例模式的设置循环依赖

    ​ 单例模式下，构造循环依赖是无法解决的，但是设置循环依赖是可以解决的。

    单例模式的创建流程

​ 为了实现单例的提前暴露，Spring 使用了三级缓存。

  • singletonObjects：单例缓存，存储已经实例化完成的单例

  • singletonFactories：生产单例的工厂的缓存，存储工厂

  • earlySingletonObjects：提前暴露的单例缓存，此时的单例刚刚创建完，还没有进行依赖注入

    ​ 使用该 SingletonFactory 获取实例，使用的是 getEarlyBeanReference 方 法，返回一个未初始化的引用。

读取缓存的地方见 DefaultSingletonBeanRegistry

​ 先尝试从 singletonObjects 和 singletonFactory 读取，没有数据，然后尝试 singletonFactories 读取 singletonFactory，执行 getEarlyBeanReference 获取到引用 后，存储到 earlySingletonObjects 中。

​ 这个 earlySingletonObjects 的好处是，如果此时又有其他地方尝试获取未初始化的单 例，可以从 earlySingletonObjects 直接取出而不需要再调用 getEarlyBeanReference。

创建实例

​ 获取到完整的 RootBeanDefinition 后，就可以来实例化具体的 Bean。AbstractAutowireCapableBeanFactory.createBeanInstance ，返回 Bean 的包装类 BeanWrapper，一共有三种策略：

  • 使用工厂方法创建：instantiateUsingFactoryMethod
  • 使用有参构造创建：autowireConstructor
  • 使用无参构造创建：instantiateBean

注入属性

​ IOC 思想在此体现，在这一步，Spring降低了各个类之间的耦合度。

处理细节

  • 应用 InstantiationAwareBeanPostProcessor 处理器，在属性注入前后进行处 理。假设我们使用了 @Autowire 注解，这里会调用到 AutowiredAnnotationBeanPostProcessor 来对依赖的实例进行检索和注入的， 它是 InstantiationAwareBeanPostProcessor 的子类。
  • 根据名称或者类型进行自动注入，存储结果到 PropertyValues 中。
  • 应用 PropertyValues，填充到 BeanWrapper。这里在检索依赖实例的引用的 时候，会递归调用 BeanFactory.getBean 来获得。

初始化

  • 触发 Aware：

    • BeanFactoryAware：用来获取 BeanFactory

    • ApplicationContextAware：用来获取 ApplicationContext

    • ResourceLoaderAware：用来获取 ResourceLoaderAware。

    • ServletContextAware：用来获取 ServletContext。

      Spring 在初始化阶段，如果判断 Bean 实现了这几个接口之一，就会往 Bean 中注入它关 心的资源。 见 AbstractAutowireCapableBeanFactory.invokeAwareMethos

  • 触发 BeanPostProcessor：

    ​ 增强操作比如打日志、做校验、属性修改、耗时检测等等。Spring 框架提供了 BeanPostProcessor 来达成这个目标。比如我们使用注解 @Autowire 来声明依赖，就是 使用 AutowiredAnnotationBeanPostProcessor 来实现依赖的查询和注入的。

    ​ 只要 Bean 实现了 BeanPostProcessor 接口，加载的时候会被 Spring 自动识别这些 Bean，自动注册，非常方便。 然后在 Bean 实例化前后，Spring 会去调用我们已经注册的 beanPostProcessors 把处理 器都执行一遍。

    ​ 这里使用了责任链模式，Bean 会在处理器链中进行传递和处理。当我们调用 BeanFactory.getBean 的后，执行到 Bean 的初始化方法 AbstractAutowireCapableBeanFactory.initializeBean 会启动这些处理器。

  • 触发 init：

    • 实现 InitializingBean。提供了一个很好的机会，在属性设置完成后再加入自己 的初始化逻辑
    • 定义 init 方法。自定义的初始化逻辑

  • 类型转换：

    ​ Bean 已经加载完毕，属性也填充好了，初始化也完成了。 在返回给调用者之前，还留有一个机会对 Bean 实例进行类型的转换。

总结

一个 Bean 的创建离不开 获取完整定义 —> 实例化 —> 依赖注入 —> 初始化 —> 类型转换 这几步。
