展现主页信息

  当我们需要在主页展示的数据可能随时发生变化时，就不能将这些数据 硬编码到 HTML 页面中。应该从数据库中获取可用的数据并将其传递给页面，进而展现给客户。

在 Spring Web 应用中，获取和处理数据是控制器的任务，而将数据渲染到 HTML 中并在浏览器中展示则是视图的任务

为了支撑 Taco 的创建页面，需要构建一些组件

  • 用来定义taco配料属性的领域类
  • 用来获取配料信息并将其传递至视图的 Spring MVC 控制器类
  • 用来在用户的浏览器中渲染配料列表的视图模板

创建领域类

  应用的领域指的是它所要解决的主题范围：也就是会影响到对应用理解的概念和理念。

  以上就是一个非常普通的 Java 领域类，定义了描述配料所需的3个属性。可以发现，该领域类并没有 setter 、getter、equals、hashCode、toString等方法。

该领域类中使用了 Lombok 的库（它能在运行时动态的生成以上方法）

@Data 注解就是由 Lombok 提供的，它会告诉 Lombok 生成所缺失的方法，同时还会生成所有以 final 属性作为参数的构造器。

Lombok 插件可以在 getbrains 的官方插件库下载 Lombok

创建控制器类

  在 Spring MVC 框架中，控制器是重要的参与者。

  控制器主要负责处理 HHTP 请求，要么将请求传递给视图以便于渲染 HTML ，要么直接将数据写入响应体（RESTful）。

对于 Taco Cloud 应用来说，需要一个简单的控制器，来处理相应任务

  • 处理路径为 “/design” 的 HTTP GET 请求
  • 构建配料的列表
  • 处理请求，并将配料数据传递给要渲染为 HTML 的视图模板，发送给发起请求的 Web 浏览器

  对于 DesignTacoController ，先来关注类级别所应用的注解。

  • @Slf4j：这是 Lombok 所提供的注解，在运行时，它会在这个类中自动生成一个 Slf4j（Simple Logging Facada For Java） Logger。这个注解和在类中通过代码显示声名的效果一样。

  • @Controller：这个注解会将这个类识别为控制器，并且将其作为组件扫描的候选者，所以 Spring 会发现它并自动创建一个 DesignTacoController 的实例，并将该实例作为 Spring 应用上下文中的 Bean。

  • @RequestMapping：当这个注解运用到类级别的时候，它能够指定该控制器所处理的请求类型。上例中，规定了当前控制器将会处理路径以 “/design” 开头的请求。

处理 GET 请求

  修饰 showDesignForm() 方法的 @GetMapping 注解对类级别的 @RequestMapping 进行了细化。 @GetMapping 结合类级别的 @RequestMapping ，指明当接受到对 “/design” 的 HTTP GET请求时，将会调用**showDesignForm() **来处理请求。

@GetMapping 注解是在 Spring 4.3 引入的。在 Spring 4.3 之前需要用到方法级别的 @RequestMapping 注解做为替代。

@RequestMapping(method = RequestMethod.GET)

介绍下 Spring MVC 中所有可用的请求映射注解

注解 描述
@RequestMapping 通用的请求处理
@GetMapping 处理 HTTP GET 请求
@PostMapping 处理 HTTP POST 请求
@PutMapping 处理 HTTP PUT 请求
@DeleteMapping 处理 HTTP DELETE 请求
@PatchMapping 处理 HTTP PATCH 请求

在 showDesignForm() 方法中构建了一个 Ingredient 对象的列表，当然这个列表是硬编码的。

接下来，通过配料的类型过滤列表。配料类型的列表会作为属性添加到 Model 对象上，这个对象是以参数的形式传递给 showDesignForm() 方法的。

Model 对象负责在控制器和展示数据的视图之间传递数据。实际上，放到 Model 属性中的数据将会复制到 Servlet Response 的属性中，这样视图就可以找到这些数据了。

方法最后返回 “design“，就是视图的逻辑名称，会用来将模型渲染到视图上。

设计视图

  运行应用并在浏览器上访问 ”/design“ 路径，控制器中的方法就会被调用，它会从 repository 中获取数据并放到模型中，然后将请求传递给视图。如果没有定义视图，请求将会报 HTTP 404(Not Found) 的错误。

Spring 提供了多种定义视图的方式。包括 JavaServer Page(JSP)、Thymeleaf、freeMarker、Mustache 和基于 Groovy 的模板。

模板的工作原理

  类似 Thymeleaf 的视图库在设计时是与特定的 Web 框架解耦的。因此，它们无法感知 Spring 的模型抽象，所以无法与控制器放到 Model 中的数据协同工作。但是，它们可以与 Servlet 的 Request 属性协作。在 Spring 将请求转移到视图之前，它会把模型数据复制到 request 属性中，Thymeleaf 和 其它的视图模板就能访问到这些数据了。

我们为各种类型的配料重复定义**<div>**片段，并且还添加了一个 Submit 按钮，以及用户用来定义其作品名称的输入域。

由于还没有为接受创建 taco 的请求做准备。如果提交设计表单，用户就会遇到一个错误 HTTP 405（Request Method ”POST“ Not Supported）

处理表单提交

  由于视图中的表单设置成 POST,并没有声明 action 属性。这意味着当表单提交的时候，浏览器会收集表单中的所有数据，并以 HTTP POST 的方式将其发送至服务端，发送路径域渲染表单的 GET请求 路径相同，也就是 ”/design“

因此，需要一个控制器处理方法，来处理针对 ”/design“ 的 POST请求。

当表单提交时，表单中的输入域就会绑定到 Taco 对象的属性上，该对象会以参数的形式传递给 以上的控制器处理方法。之后就可以在该控制器方法中对 Taco 对象进行任意操作。

由于表单提交的数据包含多个 checkbox 元素，它们的名称都是 ingredients。还有一个 name 的文本输入元素。这些数据直接对应 Taco 类的 ingredients 和 name 属性。

  当前的控制器处理方法同样返回了一个 String 类型的值，这个值同样代表了一个要展现给用户的视图。

  但是，由于返回的值带有 ”redirect：“前缀，表明这是一个重定向视图。表示在控制器处理方法完成之后，用户的浏览器会重定向到相对路径 /orders/current。

taco 订单提交的领域对象

浏览器订单提交之后，在命令窗中关注应用的日志来查看订单信息。

Order submitted: Order(name=mofang, street=dodos, city=dsa, state=, zip=1213, ccNumber=434221, ccExpiration=, ccVV=121)

可以发现，尽管我们的控制器完成了它的工作并处理了表单提交，但是也让一些错误的信息混了进来。表单中的大部分输入域包含的可能都是不正确的信息。因此需要进行表单校验。

校验表单输入

  如果我们没有指明表单的输入域该如何进行校验，就没有什么能够阻止用户在创建 taco 的时候不选择任何配料，或则输入空的快递地址。

常用表单校验方法

  • 在控制器方法中校验：在方法中使用大量的 if/then 代码块，逐个检查，确保每个输入域都满足对应的校验规则。这样会非常繁琐，并且难以阅读和检查。
  • 使用 Spring 支持的 Java 的 Bean 校验 API： Bean Validation API，也被称为 JSR-303。我们额能够更容易地校验规则，而不必在应用程序中显示的编写声明逻辑。借助 Spring Boot，要在项目中添加校验库，我们甚至不需要做任何特殊的操作，因为 Validation API 以及 Validation API 的 Hibernate 实现将会作为 Spring Boot Web starter 的传递性依赖自动添加到项目中。

在 Spring MVC 中应用校验，需要

  • 在要被校验的类上声明校验规则。也就是 Taco 类
  • 在控制器方法中声明要进行校验。也就是 DesignTacoController 的 processDesign() 方法和 OrderController rocessOrder() 方法
  • 修改表单视图以展现校验错误

声明校验规则

需求一：对于Taco 来说，我们期望确保 name 属性不为空或null，同时期望选中的配料至少要包含一项。

  • @NotNull：该注解指明当前属性不能为null

  • @Size：该注解指明当前属性至少有多少字符，message属性定义了当输入的信息不满足声明的校验规则时要给用户展示的消息。

需求二：对于Order来说，对于地址相关的属性，只要确保用户没有提交空白字段即可，使用 Hibernate validation 的 @NotBlank 注解。支付相关的字段，不仅要确保 ccNumber 的属性不为空，还要 保证它所包含的值是一个合法的信用卡号码。ccException 属性必须符合 MM/YY 格式。ccCVV 属 性需要一个3位的数字。

  • @CreditCardNumber：这个注解声明属性的值必须是合法的信用卡号，它要能通过 Luhn 算法的检查。这能防止用户有意或无意的输入错误的数据，但是该检查并不能确保这个信用卡号真的分配给了某个账户，也不能保证这个账号能够用来进行支付。
  • @Pattern：在这个注解中，我们为其提供了一个正则表达式，确保属性值符合预期的格式。
  • @Digits：这个注解能够确保它的值包含3位有效数字。

在表单绑定时执行校验

  要想在提交至对应的控制器方法时执行对应的校验，需要为对应的控制器处理方法的对应参数（Taco/Order）上添加一个 Java Bean Validation API 的 @Valid 注解。

执行 Taco 校验

@Valid 注解会告诉 Spring MVC 要对提交的 Taco 对象进行校验，校验时机是**在它绑定完表单数据之后，调用控制器处理方法之前**。

Error对象：用来捕获校验错误的细节，然后传递给当前的控制器处理方法，进行错误处理。

执行 Order 校验

如果这两个模块都没有错误，那么控制器处理方法就会允许处理提交的数据。如果存在错误，请求就会被转发到表单视图上，以便用户有机会纠正他们的错误。

展示校验错误

  Thymeleaf 模板提供了便捷的的访问 Errors 对象的方法，借助 fields 及其 th:errors 属性。

span 标签中使用了 class 属性来为错误信息添加样式。

th:if 属性决定是否显示该元素。

fields 属性的 hasErrors() 方法会检查 ccNumber 域是否存在错误。如果存在，就会渲染 span 标签。

th:errors 属性引用了 ccNumber 的输入域，如果该输入域存在错误，那么它将会 span 元素的占位符内容替换成校验信息。

使用视图控制器

如果一个控制器非常简单，不需要填充模型或处理输入时，可以通过声明视图控制器定义控制器。

  WebConfig 类 实现了 WebMvcConfigurer 接口。该接口中定义了多个方法来配置 Spring MVC。由于它提供的方法都是默认的，所以只需要覆盖所需要的方法即可。

选择视图模板库

Spring Boot 自动配置功能所支持的模板方案

模板 Spring Boot starter 依赖
FreeMarker spring-boot-starter-freemarker
Groovy Templates spring-boot-starter-groovy-templates
Java Server Pages 无（由 Tomcat 或 Jetty 提供）
Mustache spring-boot-starter-mustache
Thymeleaf spring-boot-starter-thymeleaf

只需要选择想要使用的视图模板库，将其作为依赖项添加到构建文件中，就可以在 /templates 目录下编写模板了。

Spring Boot 会探测到你所选择的模板库，并自动配置为 Spring MVC 控制器生成视图所需要的各种组件。

如果使用 JSP模板。由于 Java Servlet 容器包括嵌入式的 Tomcat 和 Jetty 容器，通常会在 “/WEB-INF” 目录下寻找 JSP。这样的话，就只能将应用构建为 WAR 文件并部署到传统的 Servlet容器中。

模板缓存

开发期，一般都需要禁用模板缓存。只需要将相关的缓存属性设置为 false

每种模板库所对应的缓存属性

模板 启用缓存的属性
FreeMarker spring.freemarker.cache
Groovy Templates spring.groovy.template.cache
Mustache spring.mustache.cache
Thymeleaf spring.thymeleaf.cache

默认情况下，这些属性都设置成了 true，以便于启用缓存。要禁用缓存，只需要在 application.properties 中添加相关代码即可。
