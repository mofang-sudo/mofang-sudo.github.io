什么是Spring

  任何实际的应用程序都是由很多组件组成的，每个组件负责整个应用功能的

一部分，这些组件需要与其他的应用元素进行协调已完成自己的任务。当应用

程序运行时，需要以某种方式创建并引入这些组件。

  Spring的核心是提供了一个容器（Container），通常称为Spring应用上下文（Spring Application Context），它们会创建和管理应用组件。这些组件也可以称为Bean，会在Spring应用上下文中装配在一起，从而形成一个完整的应用程序。

Spring的Bean管理

把对象的创建与对象的掉用都交给Spring进行管理。降低耦合度。

将Bean装配在一起的行为是通过一种基于依赖注入（Dependency injection）的模式实现的。通常，都是通过构造器参数和属性访问方法来实现。

  • 基于一个或多个XML文件

  • 基于java的配置

  @Configuration注解会告知Spring这是一个配置类，会为Spring应用上下文提供Bean.这个配置类的方法使用**@Bean注解进行了标注，表明这些方法所返回的对象会以 bean 的形式添加到 **Spring 的应用上下文 中（默认这些Bean所对应的bean ID与定义它们的方法名称是相同的）

  相对于基于XML的配置方式，基于java的配置会带来更多额外的收益，包括更强的类型安全性以及更好的重构能力。不管使用的哪种方式，都是只有Spring不能进行自动装配的时候才是必要的。

自动装配

  在Spring技术中，自动装配（Autowired）和组件扫描（Component scanning）。借助组件扫描技术，Spring能够自动发现应用类路径下的组件，并将它们创建成Spring应用上下文中的Bean。借助自动装配技术，Spring能够自动为组件注入它们所依赖的其他Bean。

初始化Spring应用

  Spring Initializr是一个基于浏览器的Web应用，同时也是一个RESET API，能够生成一个Spring项目结构的骨架。使用Spring Initializr有好几种方式：

  1. 通过地址为 https://start.spring.io/ 的Web应用

  2. 在命令行中使用 curl命令

  3. 在命令行中使用 Spring Boot命令行接口

  4. 在 Spring Tool Suite 中创建项目

  5. 在 IntelliJ IDEA 中创建新项目

  6. 在 NetBeans 中创建新项目

使用 IntelliJ IDEA 初始化项目

  在FIle > New菜单下选择Project菜单项。此时将会打开新 Spring Initializr 项目向导的第一页。直接点击Next按钮进入下一页。如果你想使用与 http://start.spring.io 不同的 Spring Initalizr，就可以选中Custom单选框，并输入想要使用的Spring Initalizr的基础URL。

  点击Next按钮之后，可以看到一个输入项目基本信息的页面。这个页面上有很多输入域与Maven pom.xml中的信息是一致的。实际上，在Type输入域中选择Maven Project，就能发现这些输入域的用途所在。也可以使用Gradle Project。

  在填写完必要信息后，点击Next按钮将会展现项目依赖页。

  在最左侧，依赖是按照分类来组织的。选中某一分类时，这个分类对应的可选项会显示在中间区域。已经选中的依赖将会（按照分类）列在最右侧。

  在选择完以来之后，点击Next按钮，我们将会看到项目向导的最后一页。在这个页面中，我们要输入项目的名称并指定项目要放到磁盘的什么位置。

  点击Finish按钮，项目将会创建并加载到 IntelliJ IDEA 的工作空间中。

检查Spring项目的结构

项目加载到IDE后，完整的目录结构如下

  • mvnw和mvnw.cmd：这是Maven包装器（wrapper）脚本。借助这些脚本，即便机器上没有安装Maven，也可以构建项目。
  • pom.xml：Maven的构建规范。
  • TacoCloudApplication.java：这是Spring Boot主类，它会启动该项目。
  • application.properties：这个文件起初是空的，但是它为我们提供了指定配置属性的地方。
  • static：在这个文件夹下，你可以存放任意为浏览器提供服务的静态内容（样式表、JavaScript等），该文件夹初始为空。
  • templates：这个文件夹中存放用来渲染内容到浏览器的模板文件。这个文件夹初始为空。可以往里面添加 Thymeleaf 模板
  • TacoCloudApplicationTests.java：简单的测试类，它能确保Spring应用上下文可以成功加载。

构建规范

  Spring Initializr 所生成的 pom.xml 文件包含了我们所选择的依赖

  1. <packaging：这里选择了构建成一个可执行的 JAR 文件，而不是 WAR 文件。对 Web 应用来说很奇怪。打包成 JAR 文件是 基于云思维 做的选择。尽管 WAR 文件非常适合部署到传统的 Java 应用服务器上，但对于但多数云平台来说 WAR 并不是理想的选择。但是所有的 Java 云平台都能够运行可执行的 JAR 文件。所以，Spring Initializr 默认会使用基于 JAR的打包方式。
  2. <parent：表明项目以 spring-boot-starter-parent 做为其父 POM。
  3. <dependencies：Spring Boot starter 依赖的特别之处是他们本身并不包含库代码，而是传递性的拉取其它的库。
  4. Spring Boot 插件：它提供了一个 Maven goal，允许我们使用 Maven 来运行项目。确保依赖的所有库都会包含在可执行 JAR 文件中，并且能够保证它们在运行时类路径下是可用的。它会在 JAR 文件中生成一个 manifest 文件，将引导类（也就是 TacoCloudApplication）声明为可执行 JAR 的主类。

引导应用

  由于我们是通过执行 JAR 文件的形式来运行应用，所以要有一个主类，它会在 JAR 运行时被执行。还需要一个最小化的 Spring 配置，以引导该应用。

@SpringBootApplication 注解明确表明这是一个 Spring Boot 应用，它非常强大，是一个组合注解，组合了 3 个其它的注解。

  • @SpringBootConfiguration：将该类声明为配置类。这个注解是 @Configuration 注解的特殊形式。
  • @EnableAutoConfiguration：启用 Spring Boot 的配置。这个注解会告诉 Spring Boot 自动配置它认为我们会用到的组件。
  • @ComponentScan：启用组件扫描。Spring 会自动发现有注解的组件将它们注册为 Spring 应用上下文中的组件。

测试应用

虽然这个类只有一个空的测试方法，但是该测试类还是会执行必要的检查，确保 Spring 应用上下文能够成功加载。通过测试类可以反映相关问题所在。

@RunWith(SpringRunner.class) 注解：它是 JUnit 的注解，会提供一个测试执行器（runner）来指导 JUnit 如何运行测试。相当于一个应用插件，可以提供自定义的测试行为。

SpringRunner：是一个 Spring 提供的测试运行器，它会创建测试运行所需要的 Spring 应用上下文。

@SpringBootTest：会告诉 JUnit 在启动测试的时候要添加 Spring Boot 的功能。

Spring 简单应用

  首先，为项目添加一个主页。在添加主页时，将会创建两个代码构件：

一个控制器类，用来处理主页相关的请求。

一个视图模板，用来定义主页的内容

在正式运行之前，还会编写一个测试类来测试主页。

处理 Web 请求

  Spring 自带了一个强大的 Web 框架 ——> Spring MVC。核心是 控制器（Controller）的理念。控制器是处理请求并以某种方式进行信息响应的类。在面向浏览器的应用中，控制器会填充可选的数据模型并将请求传递给一个视图，以便于生成返回给浏览器的 HTML。

@Controller：目的是让组件扫描将这个类识别为一个组件。有一些其它的注解有类似的目的（**@Component、@Service、@Repository）。这里使用 **@Controller 更能描述这个组件在应用中的角色。

home()：是一个简单的控制器方法。**@GetMapping注解** 表明如果针对 “/”发送 HTTP GET 请求，那麽这个方法将会处理请求

返回的 String 类型 home 值：该值将会被解析为视图的逻辑名。

模板名称是由逻辑视图名派生而来的，再加上 “/templates” 前缀和 “.html” 后缀。最终形成的模板路径将是 “/templates/home.html”。所以要将模板放到项目的 “/src/main/resources/templates” 目录中。

定义视图

img 标签：使用了 Thymeleaf 的 th:src 属性和 @{…} 表达式，以便引用相对于上下文路径的图片。

  图片是相对于上下文的 “/images/TacoCloud.png” 路径来进行引用。类似这种静态资源都是放到 “/src/main/resources/static” 文件夹中的。

测试控制器

  当前测试中，没有使用 @SpringBootTest 注解，而是添加了 @WebMvcTest 注解。这是 Spring Boot 提供的一个特殊测试注解，它会让这个测试在 Spring MVC 应用的上下文中执行（将 HomeController 注册到 Spring MVC 中，之后就可以发送请求了）。

  @WebMvcTest：同样可以为测试 Sping MVC 应用提供 Spring 环境的支持。可以看到我们并没有启动一个服务器来进行测试。这是因为该注解仿造了 Spring MVC 的运行机制。

测试类被注入了 MockMvc，能够让测试实现 mockup

  testHomePage()：首先使用 MorkMvc 对象对 “/” 发起 HTTP GET 请求。并设置了预期结果：

  • 响应应该具备 HTTP 200（OK）状态
  • 视图的逻辑名称应该是 home
  • 渲染后的视图应该包含文本 “welcome to …”

构建和运行项目

不管采用什么方式初始化项目，都可以在命令行中使用java -jar 命令来运行应用

% java -jar demo.jar

即使不采用 JAR 文件而使用 WAR 文件来进行分发，依旧可行

% java -jar demo.war

如果是使用 Maven 或 Gradle 构建的项目，可以使用 Spring Boot Maven 或 Gradle 插件来运行应用

% mvn spring-boot:run 或 % gradle bootrun

之后，就可以在打开 Web 浏览器并访问 http://localhost:8080

虽然我们根本就没有将应用部署到 Tomcat 上，就可以访问到。只是因为 Spring Boot 内部集成了 Tomcat。

Spring Boot DevTools

  DevTools 是为 Spring 开发人员提供的便利的开发期工具。

  • 代码变更后应用会自动重启
  • 当面向浏览器的资源（模板、JavaScript、样式表）等发生变化时，会自动刷新浏览器
  • 自动禁用模板缓存
  • 如果使用 H2 数据库，内置了 H2控制台

应用自动重启

  当 DevTools 运行的时候，应用程序会被加载到 Java 虚拟机（Java Virtual Machine，JVM）两个独立的类加载器中。其中一个类加载器会加载 java 代码、属性文件以及项目中 “src/main” 路径下的所有内容。这些条目可能会经常发生变化。另一个类加载器会加载依赖的库，这些库不太可能经常发生变化。

  当探测到变更的时候，DevTools 只会重新加载博涵项目代码的类加载器，并重启 Spring 的应用上下文，另外一个类加载器和 JVM 会原封不动。这个策略减少了应用启动的时间。

  如果在构建规范中添加、变更、移除依赖项的时候，由于自动重启无法反映依赖项的变化，需要手动重启。

禁用模板缓存

  默认情况下，类似 Thymeleaf 模板方案在配置时会缓存模板解析的结果。方便在每次请求提供服务时，模板不用重新解析。但这在开发期存在弊端。

  开发期，如果缓存模板，我们刷新浏览器就无法看到模板变更的效果。即使对模板做了修改，在应用重启之前，缓存的模板依旧有效。

浏览器自动刷新

  如果连浏览器的刷新按钮都懒得点。完全 OK

  DevTools **在运行时，会和应用程序一起，同时自动启动一个 **LiveReload 服务器。服务器本身没有什么用处。但是，当于 LiveReload 浏览器插件 结合时，就能够在浏览器提供服务的所有内容发生变化时自动刷新浏览器。

  LiveReload 有针对 Chrome、Safari 和 Firxfox 的浏览器插件。LiveReload

内置的 H2 控制台

如果使用 H2 数据库进行开发，DevTools 会自动启用 H2。通过Web浏览器访问。

http://localhost:8080/h2-console

就可以看到应用所使用的数据。

Tips

  可以看到，使用 Spring 进行开发，只需关注应用需求的代码，无需考虑如何满足框架的需求（只占代码库的很少一部分）。

感受不到的框架的框架（frameworkless framework）

在 pom.xml 文件中，声明了对 Web 和 Thymeleaf starter 的依赖，这两项以来会传递进大量的其它依赖：

  • Spring 的 MVC 框架
  • 嵌入式的 Tomcat
  • Thymeleaf 和 Thymeleaf 布局方言

还引入了 Spring Boot 的自动装配库。当应用启动时，Spring Boot 的自动装配会探测到这些库，并完成相应的功能：

  • 在 Spring 应用上下文中配置 Baen 以启用 Spring MVC
  • 在 Spring 应用上下文中配置嵌入式的 Tomcat 服务器
  • 配置 Thymeleaf 视图解析器，以便于使用 Thymeleaf 模板渲染 Spring MVC视图
