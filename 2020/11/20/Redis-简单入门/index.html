Redis 启动后的杂项基础知识

单进程

​ Redis 通过单进程模式来处理客户端的请求，对读写等事件的响应通过对 epoll 函数的包装来实现的，Redis 的实际处理效率完全依靠主进程的执行效率。

​ Epoll 是 Linux 内核为处理大批量文件描述符而做了改进的 epoll，是 Linux 下多路复用 IO 接口 select/poll 的增强版本，它能显著提高程序在大量并发连接中只有少量活跃的的情况下的系统利用率。

默认数据库

​ Redis 默认有 16 个数据库，类使数组下标从零开始，初始默认是 0 号库。通过 select 指令来切换数据库。

统一密码管理

​ Redis 的所有数据库共用同一个密码，要么全部成功，要么全部失败。

​ 但是 Redis 只做缓存，不做安全。

索引

​ Redis 的索引都是从零开始

默认端口

​ Redis 的默认端口是 6379

Redis 的数据类型&常用命令

​ Redis 有5种数据类型。string（字符串）、hash（哈希）、list（列表）、set（集合）、zset（有序集合）。

Redis 命令大全

Key

String

​ String 类型是 Redis 最基础的数据类型。String 类型是二进制安全的，因此可以描述任何数据，包括 jpg 图片&序列化对象。String 类型的 value 最大可存储 512M 的数据。

List

​ List（列表）是简单的字符串列表，按照插入顺序排序。底层是 LinkedList（双向列表），既可以从头部插入也可以从尾部插入。

性能分析：如果键不存在，创建新的链表；如果键存在，存入数据；如果键中数据全部移除，键也就不存在了。

Set

​ Set（集合）是 String 类型的无序集合，底层是 HashTable。

Hash

​ Hash（哈希）相当于 Java 中的 Map。常用来存储对象

ZSet

​ ZSet（有序集合）和 Set 类型一样，都是 String 类型的集合，且不允许有重复的成员。

​ 每个元素关联一个 Double 类型的分数来实现排序。

Redis 的配置文件

Units

​ 配置大小单位，只支持 bytes 不支持 bit，对大小写不敏感。

Includes

​ 用来包含一些个性化的定制配置。

General

daemonize：开启守护进程

pidfile：指定 redis server 进程的 pid 文件路径

port：指定 redis server 启用的端口

tcp-backlog：设置tcp的backlog，backlog其实是一个连接队列，backlog队列总和=未完成三次握手队列 + 已经完成三次握手队列。默认 511

timeout：客户端空闲多长时间后断开连接，0 表示用不断开

bind：端口绑定的 ip

tcp-keepalive：每隔多长时间检测 tcp 的 keepalive，0 表示不检测，官方推荐 60s

logfile：日志文件的名字

loglevel：日志级别（debug、verbose、notice、warning），默认 notice

syslog-enable：将日志输出到 syslog 中，默认 no，开启 yes

syslog-ident：系统日志默认 redis 开头

syslog-facility：日志输出的设备，值为 user | local0 ~ local7

databases：初始数据库数量

Security

​ 设置 Redis 的访问密码。不建议设置，Redis 只做缓存不做安全。

Limits

maxClients：最大客户端连接数。设置redis同时可以与多少个客户端进行连接。默认情况下为10000个客户端。当你
无法设置进程文件句柄限制时，redis会设置为当前的文件句柄限制值减去32，因为redis会为自
身内部处理逻辑留一些句柄出来。如果达到了此限制，redis则会拒绝新的连接请求，并且向这
些连接请求方发出“max number of clients reached”以作回应。

maxMemory：分配给 Redis 的最大使用内存。一旦到达内存使用上限，redis将会试图移除内部数据，如果你的redis是主redis（说明你的redis有从redis），那么在设置内存 使用上限时，需要在系统中留出一些内存空间给同步队列缓存，只有在你设置的是“不移除”的情况下，才不用考虑这个因素

maxMemory-policy：Redis 的缓存清除策略。volatile-lru：按照 LRU 算法移除设置了过期时间的key。allkeys-lru：按照 LRU 算法移除key，针对所有的key；

​ volatile-random：随机移除过期集合中的key；allkeys-random：随即移除，针对所有的key；volatile-ttl：移除 ttl 时间最短的；noeviction：不进行移 除。默认 noeviction。

maxMemory-sample：设置样本数量，LRU算法和最小TTL算法都并非是精确的算法，而是估算值，所以你可以设置样本的大小，redis默认会检查这么多个key并选择其中 LRU的那个。

SnapShoting

save <.seconds> <.change>：RDB是整个内存的压缩过的Snapshot，RDB的数据结构，用来配置复合的快照触发条件。默认的触发条件是：

​ save 900 1 save 300 10 save 60 10000

​ 禁用RDB的持久化策略：只要不设置任何save指令，或者给save传入一个空字符串参数也可以

stop-writes-on-bgsave-error：如果配置成no，表示你不在乎数据不一致或者有其他的手段发现和控制

rdbcompress：对于存储到磁盘中的快照，可以设置是否进行压缩存储。如果是的话，redis会采用LZF算法进行压缩。如果你不想消耗CPU来进行压缩的话，可以设置为关闭 此功能。

rdbchecksum：在存储快照后，还可以让redis使用CRC64算法来进行数据校验，但是这样做会增加大约10%的性能消耗，如果希望获取到最大的性能提升，可以关闭此功能

rdbfilename：RDB持久化后的文件名称。

dir：RDB持久化后的文件保存路径

Append Only Mode

appendonly：开启 aof ，默认 no

appendfilename：AOF 的文件名

appendsync：AOF 的执行策略，默认时 everysec，执行策略：no、always、everysec

no-appendsync-no-rewrite：重写时是否可以运用 appendsync，默认 no，保证数据的安全性。

auto-aof-rewrite-precentage：设置重写的基准值，

auto-aof-rewrite-min-size：设置重写的基准值

aof-load-truncated：默认为 yes

Redis 的持久化

RDB

​ RDB（Redis Database）：在指定的时间间隔内将内存中的数据集 SnapShot 写入到磁盘中。恢复时将快照文件直接读取到内存中。

​ Redis 会单独 fork 一个子进程来进行持久化，会先将数据写入到一个临时文件中，待持久化过程结束，再将这个临时文件替换掉上一次持久化好的文件。

​ 整个过程主进程不会进行任何的 IO 操作，这就确保了极高的性能。

​ 如果需要进行大规模的数据恢复，且对数据恢复的完整性不是非常敏感，那么 RDB 方式比 AOF 方式更加高效。

​ RDB 的缺点是有可能会丢失最后一次持久化的数据。

fork：作用是复制一个与当前进程一样的进程，新进程的数据与原进程的数据一致，但是是一个全新的进程，并作为原进程的子进程。

RDB 保存的文件默认是 dump.rdb 文件

RDB快照的触发

  • 符合配置文件中的快照配置
  • 通过 save&bgsave。save 只管保存，其它全部阻塞；bgsave 会在后台异步进行快照操作，通过 lastsave 命令获取最后一次执行成功快照的时间。
  • flushAll 也会触发快照，不过 dump.rdb 文件是空的，没有意义。

恢复？ 将备份文件移动到 redis 的安装目录，并启动服务。

优点？ 适合大规模的数据回复。对数据完整性&一致性要求不高。

缺点？ 在一定时间间隔内做一次备份，当 redis 意外宕机时，会丢失最后一次的持久化数据。fork 时内存中的数据被复制了一份，内存膨胀了两倍。

AOF

​ AOF（Append Only File）：以日志的形式来记录所有写的操作。将 redis 执行过的所有写的指令记录下来（读的指令不记录），只能追加文件不能改写文件。

​ 在 Redis 服务启动之初，会读取该文件重新构建数据，相当于重新执行一边所有指令已完成数据的恢复工作。

​ AOF 保存的文件默认是 appendonly.aof

AOF 的启动&恢复&修复：将有数据的 aof 文件保存到服务启动的目录，重启 redis 服务会自动加载。

​ 修复有异常错误的 aof 文件 Redis-check-aof –fix

AOF 的 rewrite 机制：解决 AOF 追加文件时文件越来越大的问题。当文件超过设定的阈值时，就触发 rewrite 机制，对 aof 文件进行压缩，只保留能够恢复数据的最小指令 集。使用命令：bgrewriteaof

AOF 的 rewrite 原理：当 aof 文件过大时，会 fork 一个新的进程来将文件重写（实际先写临时文件，再 rename）。新的进程会扫描主进程中的所有命令，记录每一条写的 命令，并不会读取旧的 aof 文件。与 快照类似。

AOF 的 rewrite 的触发机制：Redis 会记录上一次重写的 aof 文件大小，当 aof 文件超过记录的一倍且文件大小超过 64 MB 时触发。

AOF 的劣势：相同数据集，aof 文件比 rdb 文件大得多，数据恢复较慢。

小结

​ RDB 和 AOF 两种持久化方式可以同时存在。当 Redis 服务启动时，会优先加载 aof 文件，因为 aof 文件保存的数据集比 rdb 文件完整。

​ RDB 比 AOF 更适合做数据库备份，因为 aof 文件实在不断变化的。

因为RDB文件只用作后备用途，建议只在Slave上持久化RDB文件，而且只要15分钟备份一次就够了，只保留save 900 1这条规则。

如果Enalbe AOF，好处是在最恶劣情况下也只会丢失不超过两秒数据，启动脚本较简单只load自己的AOF文件就可以了。代价一是带来了持续的IO，二是AOF rewrite的最后将rewrite过程中产生的新数据写到新文件造成的阻塞几乎是不可避免的。只要硬盘许可，应该尽量减少AOF rewrite的频率，AOF重写的基础大小默认值64M太小了，可以设到5G以上。默认超过原大小100%大小时重写可以改到适当的数值。

如果不Enable AOF ，仅靠Master-Slave Replication 实现高可用性也可以。能省掉一大笔IO也减少了rewrite时带来的系统波动。代价是如果Master/Slave同时倒掉，会丢失十几分钟的数据，启动脚本也要比较两个Master/Slave中的RDB文件，载入较新的那个。

Redis 的事务

​ 一次性执行多条命令，本质上是一组命令的集合。一个事务中的所有命令都会被序列化，按照顺序串行化的执行而不会被其他命令插入，不允许加塞。

​ 一个队列中，一次性、顺序性、排它性的执行一串命令。

常见的事务

事务命令

Redis 事务执行时遇到编译时异常，全体连坐，所有命令都不执行。遇到运行时异常，冤有头债有主，除了异常命令外，都正常执行

Redis 事务的执行流程：1、开启事务；2、将多个操作命令加入到事务队列中，这些命令不会立即执行；3、串行化执行事务队列。

Redis 事务相对于传统数据库的特点：1、没有隔离级别的概念；2、不保证原子性，Redis 同一个事物如果有一条命令执行失败，其他命令并不会全部回滚，而是正常执行。

Watch 监控

悲观锁：对一切操作都很悲观，认为自己每次去拿数据的时候都认为有别人进行修改。所以每次拿数据的时候都会上锁，这样别人来拿数据就会阻塞（block），直到它拿到 这个锁。传统数据库中也有类似的机制：行锁、表锁、读锁、写锁等…

乐观锁：对一切操作都很乐观，认为自己每次去拿数据时都不会有人修改数据。但是在更新数据时会进行判断是否有人修改了数据（通过版本号），如果有人进行了修改， 则此次更新操作全部放弃。乐观锁适用于多读的情况，这样可以提高吞吐量。

CAS：check and set，Redis 通过 CAS 机制实现了 watch 监控的乐观锁机制。当 watch 监控的数据被修改时，exec 执行的事务全部放弃。

一旦执行了 exec，则之前加的监控锁全部取消掉。

Redis 的发布订阅

发布订阅：是进程的一种消息通信模式。发送者（pub）发布消息，订阅者（sub）接收消息。

​ 相当于 B站的关注或者 YouTuBe 的频道订阅，只要订阅了发布者，就能接收到发送者的消息推送。

Redis 虽然可以用来做消息中间件，但是一般不建议使用。有更成熟的产品：ActiveMQ

Redis 的主从复制

​ 主机数据发生更新后根据配置和策略，自动同步到备机的 master/slave 机制，便于容灾恢复。Master 以写为主，Slave 以读为主，实现读写分离。

具体操作

配从不配主。slave 每次与 master 断开连接后都需要重新手动配置，除非在 redis.conf 文件中进行了自动连接配置。

开启多个从库配置：1、复制多个 redis.conf 文件；

​ 2、修改 redis.conf 的配置（1、开启 daemonize：yes；2、修改 pidfile；3、修改 port；4、修改 logfile；修改dump.rdb文件名）

一主两仆：一个主库连着两个从库。从库开启时从主库复制的数据是全量复制，之后从切入点开始都是增量复制。

​ 从库只具有读的权限，没有写的权限。

​ 主库宕机后，从库不会上位，会一直原地待命。

薪火相传：上一个 slave 可以是下一个 slave 的 master。slave 同样可以接收其它 slave 的连接和同步请求，这样可以减轻主库的写压力。

​ 当 slave 变更 master 时，会清除当前库中的数据，重新从新的 master 上全量复制。

反客为主：当主库宕机时，可以使从库停止与其它库的同步，转为主库。

​ slaveof no one

主从复制的原理：slave 启动成功连接到 master 时发送一条 sync 命令

​ master 接收到 sync 命令，fork 一个子进程作为存盘进程。同时收集接收到的修改数据的命令，存盘进程把这些命令保存到文件中，存盘进程执行完毕后， master 将数据文件发送给 slave，完成此次同步。

​ 全量复制：slave 接收到数据文件，将其存盘并加载进内存中。

​ 增量复制：master 将新的修改数据集的命令发送给 slave ，完成下一次同步。

哨兵模式

​ 相当于反客为主的自动化，能够后台监控主库的状态，当主库宕机后，通过投票数选出从库上位，其余从库自动变更依赖主库。

开启哨兵模式：1、创建 sentinel.conf 文件，编写 sentinel monitor 监控的主库名字（自定义） 主库ip 主库port 投票数（1）

​ 2、启动哨兵，redis-sentienl sentinel.conf文件所在的绝对路径

​ 3、当监控的主库宕机后，会在从库中投票选举出票数最高的作为主库

​ 4、当宕机的 master 恢复后，作为单独的主库存在

一组 sentinel 能同时监控多个 master。

复制延时问题：由于所有的写操作都是先在Master上操作，然后同步更新到Slave上，所以从Master同步到Slave机器有一定的延迟，当系统很繁忙的时候，延迟问题会更加 严重，Slave机器数量的增加也会使这个问题更加严重。

Redis 的 Java 客户端 Jedis

Jedis 是 Java 用来与 Redis 交互的 API。使用时导入 Jedis 的依赖即可。

Redis 中能完成的操作，在 Jedis 中都有对应的 method 实现。

JedisPool
